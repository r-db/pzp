# Prime Zero Protocol Specification v1.0.0

> The machine-native web. Websites built for synthetic intelligence.

## Overview

Prime Zero Protocol (PZP) defines a standard structure for websites designed exclusively for synthetic intelligence consumption. PZP sites exist as a parallel layer to human-facing websites—no UI, no CSS, no visual complexity. Pure, token-efficient, machine-readable content.

## Core Principles

1. **Zero Overhead** - Direct content delivery without HTML/CSS/JS
2. **Predictable Structure** - Standard file paths and naming conventions
3. **Token Efficiency** - Maximum information per token
4. **Self-Describing** - Machines understand machines

## File Structure

```
pzp.example.com/
├── index.md              # Entry point
├── manifest.json         # Site metadata
├── context/
│   ├── essential.md      # Minimum context (~500 tokens)
│   ├── full.md           # Complete context
│   └── {topic}.md        # Topic-specific context
├── actions/
│   ├── index.json        # Action catalog
│   └── {action}.md       # Action documentation
├── data/
│   ├── schema.json       # Data schemas
│   └── {resource}.json   # Data files
└── meta/
    ├── version.json      # Version info
    ├── changelog.md      # Changes
    └── health.json       # Service status
```

## Required Files

### index.md

Entry point for agents. Must contain:
- Site name (H1)
- One-line description (blockquote)
- Identity block (domain, type, version, updated)
- Purpose statement
- Quick start navigation

### manifest.json

Machine-readable site metadata. See `schemas/manifest.schema.json`.

Required fields:
- `version` - Semantic version
- `name` - Site name
- `description` - One-line description
- `type` - service|documentation|data|api|knowledge
- `updated` - ISO 8601 timestamp
- `endpoints` - Path mappings

### context/essential.md

Minimum viable context for an agent to understand and use the site.
Target: ~500 tokens.

Must contain:
- What This Is (single paragraph)
- Key Capabilities (3-5 bullets)
- Primary Actions (table)
- Constraints (bullets)

### actions/index.json

Catalog of available actions. See `schemas/actions.schema.json`.

Each action requires:
- `name` - Unique identifier (kebab-case)
- `description` - What it does
- `endpoint` - API path
- `method` - HTTP method
- `input` - Parameter definitions
- `output` - Response schema
- `errors` - Error codes and messages

## Response Headers

All responses should include:

```http
Content-Type: text/markdown; charset=utf-8
X-Token-Count: {estimated tokens}
X-Content-Version: {version}
X-Last-Modified: {ISO 8601}
X-Context-Level: essential|full|detailed
X-PZP-Version: 1.0.0
Cache-Control: public, max-age=3600
```

## Content Negotiation

Support multiple formats:

```http
Accept: text/markdown     → Markdown response
Accept: application/json  → JSON response
Accept: text/plain        → Plain text response
```

## Discovery

### DNS TXT Record
```
_pzp.example.com TXT "v=pzp1 endpoint=pzp.example.com type=service"
```

### robots.txt
```
User-agent: *
Allow: /
X-PZP-Site: true
X-PZP-Manifest: /manifest.json
X-PZP-Version: 1.0.0
```

### HTML Link (on human site)
```html
<link rel="pzp-site" href="https://pzp.example.com/">
```

## Token Efficiency Guidelines

**Do:**
- Use tables for structured data
- Use hierarchical detail levels
- Include token count estimates
- Prefer abbreviations where clear

**Don't:**
- Repeat information across files
- Include unnecessary examples
- Use decorative language
- Add excessive whitespace

## Versioning

- Site versions use semantic versioning (MAJOR.MINOR.PATCH)
- Protocol version tracked in `protocolVersion` field
- Breaking changes require major version bump
- Deprecations documented in `meta/version.json`

## Schemas

JSON Schema definitions available at `/schemas/`:
- `manifest.schema.json` - Site manifest
- `actions.schema.json` - Action definitions
- `data.schema.json` - Data schema registry
- `version.schema.json` - Version/health info
